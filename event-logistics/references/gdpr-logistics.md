## Lawful Basis for Logistics Data

| Processing Activity | Lawful Basis | Notes |
|---|---|---|
| Dietary requirements for catering | Explicit consent (special category) | Health-related data — collect only if catering provided, delete within 30 days post-event |
| Accessibility needs | Explicit consent (special category) | Disability data — collect minimum needed, delete within 30 days post-event |
| Attendee badge data (name, company) | Contract performance | Necessary for event delivery |
| Badge scanning / access tracking | Legitimate interest or consent | Depends on granularity — room-level tracking requires consent |
| Vendor contact information | Legitimate interest | Business contacts for event delivery |
| Staff/volunteer personal data | Contract or legitimate interest | Scheduling, emergency contacts |
| CCTV / event photography | Legitimate interest with opt-out | Signage required, opt-out badges available |
| Emergency contact information | Vital interests or legitimate interest | Collect from staff, optional for attendees |
| Incident reports | Legal obligation or legitimate interest | May contain personal data — restrict access |

## Special Category Data

Dietary requirements and accessibility needs are special category data under GDPR (Article 9) because they can reveal health conditions or disabilities.

**Rules for dietary data:**
- Collect via registration form with explicit consent checkbox
- Share with catering vendor ONLY — not with sponsors, other vendors, or attendees
- Aggregated dietary counts (e.g., "12 vegetarian, 3 gluten-free") can be shared without consent
- Individual dietary details shared with caterer must be on a need-to-know basis
- Delete individual dietary data within 30 days post-event
- Never print dietary requirements on badges or visible name cards

**Rules for accessibility data:**
- Collect via registration form with explicit consent checkbox
- Share with venue and relevant vendors (AV for hearing loops, transport for wheelchair access)
- Provide alternatives to form collection (dedicated email or phone for sensitive requests)
- Delete individual accessibility data within 30 days post-event
- Accessibility arrangements should be discreet — no public identification of attendees with specific needs

## Vendor Data Processing Agreements

Every vendor that processes attendee personal data must sign a Data Processing Agreement (DPA) before receiving any data.

**Vendors that typically need DPAs:**
- Registration platform provider
- Badge printing / scanning vendor
- Catering company (receives dietary data)
- AV / streaming vendor (records sessions with attendees visible)
- Event app provider (tracks attendee activity)
- Security company (processes badge scans, CCTV)
- Photographer / videographer (captures attendee images)
- Transport / shuttle service (receives attendee names for manifests)

**DPA minimum requirements:**
- Purpose and scope of data processing
- Types of personal data processed
- Duration of processing
- Obligation to follow controller instructions
- Security measures implemented
- Sub-processor notification obligations
- Data return or deletion upon contract end
- Audit rights for the event organizer

**Venue-specific considerations:**
- Venue staff who handle registration or access control need data protection briefing
- Venue CCTV systems may capture attendee data — clarify who is data controller
- Venue WiFi that collects device data may require separate privacy notice

## On-Site Data Handling

**Badge scanning and access tracking:**
- Inform attendees at registration that badges may be scanned for access control
- Session-level tracking (which rooms an attendee visited) requires explicit consent
- Sponsor-area badge scanning requires separate, unbundled consent per GDPR — capture at registration or at the sponsor booth, never blanket opt-in. For sponsor-side lead handling, use the event-sponsors skill.
- Data from access tracking: anonymize within 30 days, retain only aggregated statistics

**Event photography and CCTV:**
- Display signage at all entrances: "This event is being photographed/recorded. If you do not wish to be photographed, please collect a [colored] lanyard at registration."
- Provide opt-out mechanism (distinct lanyard color or badge sticker)
- Brief photographers on opt-out identifiers
- Do not publish identifiable photos of attendees with opt-out indicators
- Group photos in public areas: legitimate interest with opt-out — not individual consent

**WiFi and event apps:**
- Event WiFi that tracks devices requires notice in privacy policy
- Event apps that track location or session attendance require in-app consent
- Analytics from apps should be aggregated, not individual-level

## Staff and Volunteer Data

- Collect only what is needed: name, phone, email, emergency contact, shift schedule
- Emergency contacts: collect from all on-site staff, store securely, delete 30 days post-event
- Staff photos for identification: consent required if published beyond internal use
- Volunteer agreements should include data processing clause
- Background checks (if required): separate consent, strict access control, immediate deletion after clearance

## Retention Periods

| Data Type | Retention | Notes |
|---|---|---|
| Dietary requirements | 30 days post-event | Special category — delete promptly |
| Accessibility requests | 30 days post-event | Special category — delete promptly |
| Badge scan / access logs | 30 days post-event | Anonymize to aggregated statistics |
| Vendor contact data | Duration of contract + 12 months | Business contact data |
| Staff/volunteer personal data | Event date + 3 months | Delete unless recurring engagement |
| Emergency contact data | 30 days post-event | No longer needed |
| Incident reports (with personal data) | Event date + 36 months | Legal retention for liability |
| Event photography (identifiable) | 12 months | Re-consent or delete |
| CCTV footage | 30 days | Standard retention per most EU guidelines |

## Logistics GDPR Checklist

**Pre-Event (T-3 months):**
- [ ] DPAs signed with all vendors handling personal data
- [ ] Venue data processing responsibilities clarified (controller vs. processor)
- [ ] Privacy signage ordered for all entrance points
- [ ] Opt-out mechanism defined (lanyard color, badge sticker)
- [ ] Dietary and accessibility data collection includes explicit consent

**Pre-Event (T-1 week):**
- [ ] Photographers briefed on opt-out identifiers
- [ ] Staff briefed on data handling (no attendee lists left visible, secure badge storage)
- [ ] Vendor data access restricted to minimum necessary
- [ ] Emergency contact list compiled and stored securely

**Event Day:**
- [ ] Privacy signage displayed at all entrances
- [ ] Opt-out lanyards/stickers available at registration
- [ ] Badge scanning purposes explained at each scan point
- [ ] Dietary information shared with catering in sealed/secure format

**Post-Event (T+30 days):**
- [ ] Special category data (dietary, accessibility) deleted
- [ ] Badge scan logs anonymized
- [ ] Emergency contact data deleted
- [ ] Staff/volunteer data retained only if needed for recurring event
- [ ] Vendor DPAs reviewed for data return/deletion obligations
